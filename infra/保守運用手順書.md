# SaaS管理ツール サーバーチーム向け保守・運用手順書

> **対象者:** サーバー管理チーム
> **用途:** 本番環境（Azure）の日常運用・保守・障害対応
> **アプリ担当:** 情シス（担当者さん）
> 作成日: 2026年4月

---

## 1. 構成の概要（サーバーチーム向け）

サーバーチームが管理するAzureリソースは以下の通りです。アプリケーションのコードや機能は情シス（担当者さん）が管理します。

| リソース | サービス名 | リソースグループ | 役割 |
|---|---|---|---|
| Container App | `app-dxceco-poc` | `rg-dxceco-poc` | アプリ本体（Rails） |
| Container Apps 環境 | `cae-dxceco-poc` | `rg-dxceco-poc` | コンテナ実行環境 |
| PostgreSQL | `psql-dxceco-poc` | `rg-dxceco-poc` | データベース（常時起動） |
| Container Registry | `acrdxcecopoc` | `rg-dxceco-poc` | Dockerイメージ保管 |
| Key Vault | （Key Vault名） | `rg-dxceco-poc` | シークレット管理 |
| VNet | （VNet名） | `rg-dxceco-poc` | 閉域ネットワーク |
| Azure Monitor | Log Analytics | `rg-dxceco-poc` | ログ・アラート |

### コンテナの動作モード

- **平常時:** コンテナはmin 0（停止状態）。管理者がアクセスすると自動起動（コールドスタート: 約30秒〜1分）
- **毎月1日 6:00〜8:00 JST:** CRONスケーリングで自動起動し、月次バッチを実行後に自動停止
- **データベース:** 常時起動（停止・再起動は原則不要）

## 3. 毎月の定期作業

### 3-1. 月次バッチ実行確認（毎月1日 8:00以降）

毎月1日 6:00に以下のバッチが自動実行されます。8:00以降に実行結果を確認してください。

| バッチ | 内容 |
|---|---|
| Entra IDユーザー同期 | 社内ユーザー情報をAzure ADから取得 |
| 退職者アカウント検出 | 退職後もSaaSを使用中のアカウントを検出・通知 |
| 契約更新チェック | 30日以内に更新が必要なSaaS契約を通知 |
| SaaSアカウント同期 | Entra IDエンタープライズアプリから同期 |

**確認方法（Azureポータル）:**

1. [Azure Portal](https://portal.azure.com) にログイン
2. **コンテナー アプリ** → `app-dxceco-poc` → **監視** → **ログ**
3. 以下のKQLクエリを実行し、エラーが出ていないか確認する

```kql
ContainerAppConsoleLogs_CL
| where TimeGenerated > ago(4h)
| where Log_s contains "ERROR" or Log_s contains "batch" or Log_s contains "Job"
| order by TimeGenerated desc
| take 50
```

4. `ERROR` が出ていない場合は正常終了。出ている場合は情シス（担当者さん）に連絡する

**確認方法（CLI）:**

```bash
az containerapp logs show \
  --name app-dxceco-poc \
  --resource-group rg-dxceco-poc \
  --follow
```

## 4. 監視・アラート対応

### 4-1. 設定済みアラート

以下のアラートが設定されており、指定のメールアドレスに通知が届きます。

| アラート名 | 条件 | 重大度 |
|---|---|---|
| コンテナ再起動 | 5分間で再起動が3回以上 | Sev2（警告） |
| HTTP 5xxエラー | 5分間で5xxレスポンスが10件以上 | Sev1（重大） |
| DB接続エラー | ログに `PG::ConnectionBad` を検出 | Sev1（重大） |

### 4-2. アラート受信時の初動対応フロー

```
アラートメールを受信
    ↓
Azureポータルでログを確認（手順 → 5章）
    ↓
    ├─ 自然回復している（再起動後に正常稼働）→ 担当者さんに経緯を共有するのみ
    │
    └─ 継続してエラーが出ている → 担当者さんに連絡（下記のログを添付）
```

**担当者さんへ連絡する際に共有してほしい情報:**

- アラートメールの内容（スクリーンショット）
- Azureポータルのログ（エラー周辺20行程度）
- 発生時刻と現在の状況（回復済み / 継続中）

## 5. ログ確認手順

### 5-1. Azureポータルでのログ確認

1. Azure Portal → **コンテナー アプリ** → `app-dxceco-poc`
2. 左メニュー **監視** → **ログ ストリーム**（リアルタイム確認）
3. または **監視** → **ログ**（過去ログ検索）

**よく使うKQLクエリ:**

```kql
-- 直近1時間のエラーログ
ContainerAppConsoleLogs_CL
| where TimeGenerated > ago(1h)
| where Log_s contains "ERROR"
| order by TimeGenerated desc
| take 50

-- 直近のログ（全件）
ContainerAppConsoleLogs_CL
| order by TimeGenerated desc
| take 100
```

### 5-2. CLIでのログ確認

```bash
# リアルタイムログ（Ctrl+Cで終了）
az containerapp logs show \
  --name app-dxceco-poc \
  --resource-group rg-dxceco-poc \
  --follow

# 直近100行を表示
az containerapp logs show \
  --name app-dxceco-poc \
  --resource-group rg-dxceco-poc \
  --tail 100
```

## 6. よくある障害と対応手順

### 6-1. コンテナが起動しない

**症状:** アプリURLにアクセスしても503エラーが返る

**確認手順:**

```bash
# リビジョンの状態確認
az containerapp revision list \
  --name app-dxceco-poc \
  --resource-group rg-dxceco-poc \
  -o table

# ログ確認
az containerapp logs show \
  --name app-dxceco-poc \
  --resource-group rg-dxceco-poc \
  --follow
```

**よくある原因（担当者さんに伝えてほしい内容）:**

| ログに表示されるキーワード | 原因 |
|---|---|
| `ActiveSupport::MessageEncryptor` / `credentials` | RAILS_MASTER_KEYの問題 |
| `PG::ConnectionBad` / `could not connect` | DB接続エラー |
| `FATAL: database` | DB設定の問題 |

→ **対応は情シス（担当者さん）が行います。ログをコピーして連絡してください。**

---

### 6-2. データベースに接続できない

**症状:** ログに `PG::ConnectionBad` が繰り返し出る

**確認手順（Azureポータル）:**

1. Azure Portal → **Azure Database for PostgreSQL フレキシブル サーバー** → `psql-dxceco-poc`
2. **概要** でサーバーの状態が「利用可能」であることを確認
3. **設定** → **ネットワーク** でファイアウォール規則を確認
   - 「Azure 内の任意の Azure サービスにパブリック アクセスを許可する」が **有効** になっているか

**DBサーバー自体がダウンしている場合:**

1. Azure Portal → `psql-dxceco-poc` → **概要** → **再起動** ボタンをクリック
2. 再起動後（約2〜3分）にアプリのログを確認する

---

### 6-3. SSOログインができない

**症状:** ログイン画面でEntra IDログインを試みるとエラーになる

**確認手順:**

1. Azure Portal → **Microsoft Entra ID** → **アプリの登録** → 対象アプリ
2. **証明書とシークレット** → クライアントシークレットの **有効期限** を確認
   - 期限切れ or 30日以内の場合 → 情シス（担当者さん）に連絡

> シークレットの更新作業は担当者さんが行います。有効期限の確認のみお願いします。

---

### 6-4. 月次バッチが実行されていない（毎月1日 8:00以降に確認）

**確認手順:**

```bash
# 当日6時〜8時のログを確認
az containerapp logs show \
  --name app-dxceco-poc \
  --resource-group rg-dxceco-poc \
  --tail 200
```

CRONスケーリングの設定を確認する（Azureポータル）:

1. Azure Portal → **コンテナー アプリ** → `app-dxceco-poc`
2. **アプリケーション** → **スケーリング** → `monthly-batch` ルールが存在するか確認

→ ルールが消えている or バッチのログが見当たらない場合は情シス（担当者さん）に連絡してください。

## 7. デプロイについて（サーバーチームの対応不要）

アプリケーションのデプロイは **GitHub Actions（CI/CD）が自動実行** します。サーバーチームによる手動デプロイは原則不要です。

デプロイフロー:

```
開発者がGitHubのprodブランチにマージ
    ↓
GitHub Actionsが自動実行（約2分30秒）
    ↓
新しいDockerイメージがACRにプッシュ
    ↓
Container Appsが新リビジョンに自動更新
```

デプロイ後にアプリが正常に動作しているかの確認連絡は情シス（担当者さん）から行います。

## 8. Entra IDクライアントシークレットの更新（24ヶ月ごと）

> この作業は情シス（担当者さん）が主導します。サーバーチームは以下の確認のみお願いします。

**有効期限の確認（月次コスト確認と同時に実施推奨）:**

1. Azure Portal → **Microsoft Entra ID** → **アプリの登録** → `SaaS管理ツール`
2. **証明書とシークレット** → 有効期限の列を確認
3. **残り30日以内になったら情シス（担当者さん）に連絡**

**更新作業の流れ（情シス（担当者さん）が実施、サーバーチームはContainer Apps環境変数の更新を確認）:**

| # | 作業 | 担当 |
|---|---|---|
| 1 | Entra IDで新シークレット発行 | 担当者さん |
| 2 | Container Appsの環境変数 `ENTRA_CLIENT_SECRET` を更新 | 担当者さん |
| 3 | SSOログインが正常に動作することを確認 | 担当者さん → サーバーチームに報告 |
| 4 | 旧シークレットを削除 | 担当者さん |
