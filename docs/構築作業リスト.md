# 新システム Phase 1 構築作業リスト

> **構成**: Dataverse（業務データ）+ SharePoint（空き状況・顧客向けページ）+ Microsoft Forms（問い合わせ）+ Power Apps（管理画面）+ Power Automate（自動化）

## 全体の進め方

```
STEP 1: 環境準備・データ設計
STEP 2: Dataverse構築
STEP 3: SharePoint構築（空き状況カレンダー・顧客向けページ）
STEP 4: Microsoft Forms構築（問い合わせフォーム）
STEP 5: Power Apps構築（スタッフ管理画面）
STEP 6: Power Automate構築（自動化フロー）
STEP 7: 見積書・請求書テンプレート作成
STEP 8: kintoneデータ移行
STEP 9: テスト・リリース
```

---

## STEP 1: 環境準備・データ設計

### 1-1. Power Platform環境の準備

- [ ] Power Platform管理センターで本番環境を作成
- [ ] 開発・テスト用の環境を別途作成（本番と分離）
- [ ] デジ推メンバーに管理者権限を付与
- [ ] 現場スタッフにPower Apps Premiumライセンスを付与（Dataverseアクセスに必要）

### 1-2. SharePoint サイトの準備

- [ ] SharePoint管理センターで施設予約用のチームサイトを作成
- [ ] サイトの権限設定（社内スタッフ: 編集権限、外部: 閲覧権限）
- [ ] 外部共有の設定確認（匿名リンクまたは認証なしアクセスの可否）

### 1-3. データ設計（Dataverse）

- [ ] kintoneの既存データ構造を確認・整理
- [ ] 以下のテーブル設計を決定
  - **問い合わせテーブル**（顧客情報・希望条件・ステータス・希望サービス）
  - **見積・予約テーブル**（金額・日程・サービス内容・利用種別）
  - **顧客マスタテーブル**（取引先企業・取引先担当者）
  - **料金マスタテーブル**（平日・休日・前泊などの単価）
  - **ステータス変更履歴テーブル**（タイムライン表示用）
- [ ] テーブル間のリレーション設計（問い合わせ ↔ 顧客、問い合わせ ↔ 見積など）
- [ ] ステータス遷移の定義（問い合わせ中→見積送付→予約確定→利用済→請求済）
- [ ] **利用種別の定義**（日帰り／宿泊）— 見積メール送信の分岐判定に使用

### 1-4. データ設計（SharePointリスト）

- [ ] **会議室空き状況リスト**の設計
  - 施設（選択肢: 笠間 / 木更津）
  - 日付（日付型）
  - ステータス（選択肢: 空き / 残りわずか / 満室）
  - 残数（数値）
  - 備考（テキスト: 午前のみ等の補足）

### 1-5. 見積書・請求書テンプレート設計

- [ ] 現行レポトンの見積書レイアウトを確認
- [ ] Wordテンプレートの項目・デザインを決定
- [ ] 差し込みフィールドの定義（顧客名・日程・金額・利用種別など）
- [ ] 請求書テンプレートの項目・デザインを決定

## STEP 6: Power Automate構築（自動化フロー）

### 6-1. 問い合わせ受付フロー

```
トリガー: Microsoft Forms「新しい応答が送信されたとき」
    ↓
アクション1: Forms「応答の詳細を取得する」
    ↓
アクション2: Dataverse「行を追加する」（問い合わせテーブルにレコード作成）
    - ステータス: 「問い合わせ中」
    - 利用種別: 希望サービスに「宿泊」が含まれるか判定
    ↓
アクション3: Outlook「メールの送信 (V2)」（お客様へ受付確認メール）
    - 宛先: フォーム入力のメールアドレス
    - 件名: 【施設名】お問い合わせを受け付けました（受付番号: XXXX）
    ↓
アクション4: Teams「チャットまたはチャネルにメッセージを投稿する」
    - 担当者へ通知（受付番号・会社名・施設・日程・人数・利用種別）
    - Power Apps管理画面の該当レコードへのリンク
```

- [ ] フローを作成
- [ ] Formsトリガーの設定（対象フォームを指定）
- [ ] Forms応答の詳細を取得するアクションを追加
- [ ] Dataverseへのレコード作成アクションを追加
- [ ] **利用種別の自動判定ロジック**を実装（`if(contains(希望サービス, '宿泊'), '宿泊', '日帰り')`）
- [ ] 受付確認メール送信アクションを追加（メールテンプレート作成）
- [ ] Teams通知アクションを追加（通知先チャネルまたは個人を指定）
- [ ] テスト実行・動作確認

### 6-2. 見積承認フロー

```
トリガー: Power Apps「承認」ボタン押下（Power Appsトリガー）
    ↓
アクション1: 承認「承認を作成する」（Teams通知で承認/却下の選択）
    - 承認者: 上長等
    ↓
条件分岐: 承認結果
    ├── 承認 → 6-3（見積書自動生成・送付フロー）を起動
    └── 却下 → ステータスを「却下」に更新、担当者へTeams通知
```

- [ ] フローを作成
- [ ] Power Appsトリガーの設定
- [ ] 承認コネクタで承認フローを作成
- [ ] 承認時の後続フロー呼び出しを設定
- [ ] 却下時のステータス更新・通知を設定
- [ ] テスト実行・動作確認

### 6-3. 見積書自動生成・送付フロー（⭐ 日帰り/宿泊分岐あり）

```
トリガー: 6-2（見積承認フロー）から呼び出し
    ↓
アクション1: Dataverse「行を取得する」（問い合わせ・見積・料金マスタ）
    ↓
アクション2: Word Online「テンプレートを使用してドキュメントを作成する」
    ↓
アクション3: 「ファイルの変換」（Word → PDF）
    ↓
アクション4: Dataverse「ノートに添付ファイルをアップロードする」（PDF保存）
    ↓
★ 条件分岐: 利用種別 = 「宿泊」か？
    │
    ├── 【日帰り】→ Outlook「メールの送信 (V2)」
    │     - PDF添付して自動送信
    │     - ステータスを「見積送付」に自動更新
    │     - Teams通知（送付完了）
    │
    └── 【宿泊】→ Outlook「下書きの作成 (V2)」
          - PDF添付した下書きメールを作成
          - Teams通知（「宿泊部屋の空き確認 → 下書き確認・送信してください」）
          - ステータスは「見積作成済（送信待ち）」に更新
```

- [ ] フローを作成
- [ ] Dataverseからデータ取得アクションを追加
- [ ] Wordテンプレート差し込みアクションを追加
- [ ] PDF変換アクションを追加
- [ ] Dataverseへの添付ファイル保存アクションを追加
- [ ] **条件分岐を追加**: `利用種別 eq '宿泊'`
- [ ] **日帰りルート**: Outlook「メールの送信 (V2)」を追加
  - [ ] 宛先・件名・本文・添付ファイルの設定
  - [ ] ステータスを「見積送付」に自動更新するアクション
  - [ ] スタッフへTeams通知（送付完了）
- [ ] **宿泊ルート**: Outlook「下書きの作成 (V2)」を追加
  - [ ] 宛先・件名・本文・添付ファイルの設定
  - [ ] ステータスを「見積作成済（送信待ち）」に更新するアクション
  - [ ] スタッフへTeams通知（「aiPassで宿泊部屋の空きを確認 → Outlook下書きを確認・送信してください」＋下書きへのリンク）
- [ ] テスト実行（日帰りケース）
- [ ] テスト実行（宿泊ケース）

> **💡 宿泊時に一旦停止する理由**: 宿泊部屋の空き状況はaiPass（PMS）で管理しており、Phase 1ではシステム連携がないためリアルタイムでの目視確認が必要。Phase 3のaiPass連携で自動化予定。

### 6-4. 宿泊見積メール送信後のステータス更新

担当者がOutlookの下書きを手動送信した後、ステータスを「見積送付」に更新する方法:

**方法A: 手動更新（推奨・Phase 1）**
- [ ] Power Apps管理画面にステータス変更ボタンを追加（「見積送信済みにする」）
- [ ] 担当者がメール送信後にボタン押下でステータス更新

**方法B: 半自動（将来検討）**
- Power Automateで「メールが送信済みフォルダに移動したとき」トリガーを使用
- 件名に受付番号を含めることで該当レコードを特定
